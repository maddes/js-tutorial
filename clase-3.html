Condiciones! (y bugs)

'Programar' consiste basicamente en tomar datos, tranformarlos, y devolverlos transformados.

Por ejemplo, más adelante veremos que podemos tomar una lista y ordenarla alfabéticamente, re útil no?

Ahora, no siempre tendremos certeza de que cornos el usuario nos va a dar como 'input', por lo tanto, tenemos que verificar:

		var sexo = prompt('Tu sexo:');
		if (sexo == 'femenino') {
			document.body.style.backgroundColor = 'pink'
		}
		else {
			if (sexo == 'masculino') {

			}
			else {
				var imagen = document.createElement('img')
				imagen.src =  'http://goo.gl/jk1Da0'

Bueno, como verás, esto no es super perfecto.
Podés poner mujer, varón o nada y aún así te acusan.
A esto se le suele llamar bugs, que son comportamientos inesperados en el código.

Todos los programas tienen bugs. Todos.
Sólo que algunos tienen muy muy pocos y ni se nota.

Algunos bugs, son más difíciles de encontrar que otros.
Y algunos bugs, son más dfíciles de resolver que otros.

También hay otras cosas que se les suele decir bugs, pero que en realidad no son un comportamiento inesperado sino simplemente código ineficiente.

De lo que vimos en el ejemplo anterior de condiciones, podemos decir que la estructura general es: 

		if (condicion) {
			código a ejecutar
			si la condición se cumple
		} else {
			código a ejecutar
			si la condición NO se cumple
		}

Hay una versión simplificada de esto, que NO se recomienda en general, pero sirve cuando el código a ejecutar es simple (una sola sentencia) 

		if (condicion) sentencia a ejecutar cuando la condicion se cumple
		else sentencia a ejecutar cuando la condicion NO se comple

Por supuesto, se pueden combinar, así: 

		if (condicion) {
			código a ejecutar
			si la condición se cumple
		} 
		else sentencia a ejecutar cuando la condicion NO se cumple
		
		if (condicion) sentencia a ejecutar cuando la condicion se cumple
		else {
			código a ejecutar
			si la condición NO se cumple
		}

Se pone complicado no?
Eso no es nada vieja.
En el ejemplo del comienzo de esta lección, existe lo que llamamos anidamiento. Eso significa que hay un bloque de código dentro de otro bloque de código.
En ese caso, tenemos un segundo if dentro del primer else.
(Viste? ya empezamos a hablar en Klingón y nos entendemos...)

Combinando los condicionales con y sin llaves, podemos tener muchas condiciones 'encadenadas' sin que se vea feo: 

		if (nombre == 'Daniel') {
			IQ = 140
		}
		else if (nombre == 'Einstein') {
			IQ = 120
		}
		else if (nombre == 'El Brian') {
			IQ = 60
		}
		else {
			IQ = 100
		}

La versión 'fea' de eso mismo podría ser cualquiera de estas:
(Es exactamente lo mismo, pero feo) 

		if (nombre == 'Daniel') {
			IQ = 140
		}
		else {
			if (nombre == 'Einstein') {
				IQ = 120
			}
			else {
				if (nombre == 'El Brian') {
					IQ = 60
				}
				else {
					IQ = 100
				}
			}
		}
	
		if (nombre == 'Daniel') IQ = 140
		else if (nombre == 'Einstein') IQ = 120
		else if (nombre == 'El Brian') IQ = 60
		else IQ = 100

Si bien la última no se ve muy fea, cuando las condiciones y las sentencias son más complicadas, se mezcla todo y no se entiende, porque no hay suficiente espacio entre medio para distinguirlas bien.
